# Gate 1.5 Voice Consent / Clone Voice / Brush-up Architecture Review

Gate 1.5 reviews voice consent, clone voice, Brush-up, provider identifiers, storage, replay, deletion, and Store-facing disclosure before Native Minute moves deeper into provider production readiness or native packaging.

This is docs/design-only. It does not add DB schema, migrations, route contracts, provider integrations, UI, Capacitor, Vercel settings, workers, queues, or VPS infrastructure.

## Decision

- Brush-up is a v1 adoption candidate, not a someday feature.
- The safe v1 direction is script-scoped Brush-up from an explicit user-selected best take.
- Brush-up must not replace the user's default voice, mutate the script, overwrite normal model audio, or reuse best-take material across scripts.
- Initial server architecture remains Vercel Route Handler / API Route + Supabase Storage / DB.
- No VPS, EC2, dedicated server, always-on worker, or queue is introduced during Gate 1.5.

## Current Implementation Snapshot

- Voice setup already keeps app-owned `voice_consents`, `voices`, `voice-samples`, `voice-consents`, `script_audios`, and protected replay as the product source of truth.
- ElevenLabs voice clone reads app-owned sample audio server-side from `voice-samples`; the client does not send sample audio directly to ElevenLabs.
- OpenAI custom voice is entitlement-sensitive and requires provider-side consent recording, but it is not the v1 mainline voice provider.
- `speakScript` re-fetches the owned script and voice server-side, builds a cache key from provider, app voice row, script locale/content, and style preset, then stores provider bytes in the app-owned `script-audios` bucket.
- `/api/script-audio/[audioId]` and `/api/takes/[takeId]/audio` are authenticated replay/download routes. They do not expose public signed URLs.
- Audio Library pins `script_audios` as saved model audio and `takes` as saved best takes. It is a curation layer, not the canonical audio byte store.
- Progress computes best take by persisted score / reviewed time / created time / id; saved best take is separate user curation.

## Brush-up v1 Minimal Scope

Brush-up v1 should be the smallest vertical slice that proves the loop:

1. User practices a fixed script and has a reviewed take.
2. User explicitly chooses a saved best take or current review take as Brush-up material.
3. App asks for script-scoped consent to use that take audio as voice material for that script only.
4. Server-side service re-fetches the owned script, take, recording object, review summary, weak words, coach feedback, and active provider state.
5. Server-side provider adapter creates or selects a script-scoped Brush-up voice variant.
6. Server-side service generates a Brush-up model audio candidate for the same script.
7. Generated bytes are normalized into app-owned `script-audios` storage and replay, never provider direct URLs.
8. Listen can play the Brush-up candidate as an explicit mode or saved model audio candidate.
9. User returns to record / evaluate / review / progress.

v1 should not include:

- global voice training from all recordings,
- automatic use of any best take without explicit consent,
- default voice replacement,
- cross-script reuse of Brush-up voice material,
- in-place script edit,
- provider direct URL replay,
- client-to-provider audio upload,
- background worker / queue dependency,
- multiple competing Brush-up candidates per script unless the data model explicitly supports them.

## Consent Requirement

Best take audio is user voice material. Sending it to a provider for Brush-up requires a consent distinct from normal playback and distinct from generic voice setup.

Minimum consent text must cover:

- the selected take audio will be used as provider voice material,
- the use is limited to the selected script,
- the provider name is disclosed,
- the generated Brush-up voice/audio may be derived from the user's voice,
- the user can revoke Brush-up consent for that script,
- revocation affects Brush-up artifacts but does not automatically delete the original take/review unless account deletion or a future take-deletion feature says so,
- account deletion must remove provider-side Brush-up voices, app-owned Brush-up source material if copied, generated Brush-up model audio, and related DB rows.

Implementation implication: a future schema should track Brush-up consent separately from setup voice consent. Overloading current `voice_consents` would blur global/default voice setup and script-scoped Brush-up consent.

## Script-Scoped Voice Material / Variant

Brush-up should introduce a script-scoped voice variant concept.

Recommended future model:

- `script_voice_variants` or `brushup_voice_variants`
- `user_id`
- `script_id`
- `source_take_id`
- `brushup_consent_id`
- `provider`
- server-only `provider_voice_id`
- `status`
- `created_at`
- `revoked_at`
- `provider_deleted_at`
- safe metadata only

Do not use `voices.is_default=true` for Brush-up. The existing `voices` table represents user/default voice setup for normal listen generation. Brush-up voice variants are narrower: they are derived from a take, scoped to one script, and revocable without changing normal voice setup.

## Voice Priority

Normal Listen mode:

1. Existing saved model audio or normal cached model audio for the selected script/voice/style.
2. Normal default cloned voice from `voices`.
3. Standard/built-in provider fallback only if a future provider path explicitly supports it.

Brush-up mode:

1. Accepted Brush-up model audio candidate for the current script.
2. Existing Brush-up candidate for the same script and same source consent, if cache identity matches.
3. Normal default cloned voice audio as fallback only after user chooses to leave Brush-up mode.

Brush-up voice must never become the global default voice implicitly.

## Generated Audio / Cache / Replay

Generated Brush-up audio should use the same app-owned replay principle as normal model audio:

- provider output is normalized server-side into Supabase Storage,
- user-facing playback goes through authenticated app routes,
- provider direct URLs are not persisted or shown,
- raw provider responses and provider voice ids are not returned to the client.

Existing `script_audios` may be usable for the generated audio bytes only if the cache identity can unambiguously separate normal model audio from Brush-up audio.

Future cache discriminator options:

- `generation_purpose=normal | brushup`
- `source_take_id` or source take hash
- `script_voice_variant_id`
- `brushup_consent_id`
- provider/model/style fields already used by normal generation

If this cannot be represented without confusing existing cache semantics, add a dedicated Brush-up candidate table that references a generated `script_audios` row.

## Deletion And Revoke

Consent revoke for one script should:

- stop future Brush-up generation from that take/material,
- delete or disable provider-side script-scoped Brush-up voice where provider APIs allow,
- remove app-owned copied Brush-up source material if a copy was created,
- delete generated Brush-up model audio and saved model audio pins derived from that consent, or at minimum hide them until deletion completes,
- mark related Brush-up rows as revoked/deleted with safe status only,
- keep the original take/review unless the user requests account deletion or a future take deletion feature.

Account deletion must additionally cover:

- provider-side normal cloned voices,
- provider-side Brush-up voice variants,
- `voice-samples`,
- `voice-consents`,
- `recordings`,
- `script-audios`,
- saved model audio and saved best take rows,
- Brush-up consent / variant / candidate rows,
- quota events related to voice clone, voice generation, and Brush-up generation.

Provider cleanup must use server-owned rows to derive provider ids. The client must never submit provider voice ids, storage object keys, or raw provider references for deletion.

## Future DB / Storage / API Candidates

These are future implementation candidates only:

DB:

- `brushup_consents`
- `script_voice_variants` or `brushup_voice_variants`
- `brushup_audio_candidates`
- quota event extension for `brushup_voice_creation_attempt` and `brushup_audio_generation_attempt`

Storage:

- Source best take can be read from existing `recordings` server-side.
- If transformed or copied source material is required, use app-owned storage such as `voice-samples` with a script/take scoped key, or add a dedicated `brushup-materials` bucket after policy review.
- Generated audio remains in `script-audios`.

API:

- `POST /api/scripts/[id]/brush-up/consents`
- `POST /api/scripts/[id]/brush-up/candidates`
- `GET /api/scripts/[id]/brush-up/candidates`
- `POST /api/scripts/[id]/brush-up/candidates/[candidateId]/accept`
- `POST /api/scripts/[id]/brush-up/revoke`

Route handlers should stay thin: auth, schema validation, service call, safe response. Services should re-fetch owned script/take/audio/variant rows before provider calls.

## Vercel / Supabase / Worker Decision

Vercel Route Handlers / API Routes plus Supabase Storage / DB are enough for the first Brush-up MVP if:

- one user action creates one script-scoped Brush-up candidate,
- source audio is short and already in `recordings`,
- provider clone/create voice returns within function timeout,
- generated script audio is one short script audio,
- failures can be reported synchronously or by explicit user retry,
- cleanup is operator-run or request-driven rather than scheduled background work.

Consider worker / queue / VPS only if one or more conditions become true:

- provider clone creation is async or regularly exceeds Vercel Function timeout,
- provider retry/backoff must continue after the user leaves the page,
- cleanup/revoke must be guaranteed without an operator action,
- batch deletion or retention jobs are required,
- audio conversion becomes CPU-heavy or large-file-heavy,
- concurrent Brush-up requests need dedupe/locking beyond DB constraints,
- provider latency or rate limits require durable job state,
- Store review requires stronger automated deletion proof than request-driven routes can provide.

Until those conditions are proven, do not introduce VPS, EC2, a dedicated server, an always-on worker, or a queue.

## Store / Privacy / Data Safety Impact

Brush-up increases Store-facing disclosure requirements.

Privacy / terms must explain:

- user voice recordings are processed for transcription, pronunciation evaluation, model audio generation, and optional Brush-up,
- Brush-up may send selected voice audio to the configured voice provider,
- provider names and roles,
- retention and deletion behavior,
- consent revoke behavior,
- account deletion coverage,
- support contact and expected deletion timeline.

Apple App Privacy / Google Data Safety likely need review for:

- audio / voice recordings,
- user-generated content,
- identifiers / account data,
- diagnostics or usage data if provider monitoring is added,
- AI provider processing and third-party sharing/processing,
- deletion and consent management claims.

Store reviewer notes should avoid claiming on-device-only processing. Voice material and generated audio are server/provider mediated.

## v1 vs v1.1

v1 adoption direction:

- Review-first or saved-best-take-first entry.
- One script-scoped Brush-up candidate per explicit consent.
- ElevenLabs mainline only unless OpenAI custom voice entitlement is separately approved.
- App-owned replay only.
- Revocation and account deletion paths defined before launch.

v1.1 deferral option:

- Ship Store release with normal cloned voice and Audio Library only.
- Keep Brush-up UI hidden.
- Use Gate 3.5 to implement and prove Brush-up after Store baseline approval.

Current recommendation: v1 adoption is reasonable only if Gate 3.5 is added before Capacitor/native packaging and includes schema/API/provider/deletion proof. Otherwise, defer Brush-up to v1.1 rather than shipping a partial or unclear consent model.

## Handoff To Later Gates

Gate 2:

- Draft privacy / terms / support / consent copy for Brush-up.
- Define consent revoke UX and account deletion language.
- Decide whether Brush-up-generated audio is deleted on revoke or hidden pending cleanup; conservative v1 should delete.
- Ensure App Privacy / Data Safety answers cover voice material processing.

Gate 3:

- Confirm ElevenLabs clone/create/delete behavior, latency, retry, verification-required behavior, rate limits, and cost.
- Confirm provider cleanup can delete normal cloned voices and Brush-up script-scoped voices.
- Add provider failure smoke for Brush-up source audio, provider clone creation, generated audio staging, and replay.
- Extend cost guard / kill switch / quota event design for Brush-up.

Gate 3.5:

- Implement Brush-up schema/API/service/UI after Gate 2/3 decisions.
- Prove one script -> one best take -> explicit consent -> server-side provider call -> generated app-owned audio -> listen -> record/evaluate loop.
- Prove revoke/delete path with safe evidence.
- Decide whether worker/queue is still unnecessary based on measured provider latency.

## Stop Conditions For Implementation

Stop before implementation if any of these are unresolved:

- consent copy is ambiguous,
- provider-side deletion behavior is unknown,
- provider returns async job semantics that cannot fit a Vercel request,
- best take ownership cannot be proven server-side,
- cache identity would mix Brush-up and normal model audio,
- generated audio lifecycle on revoke is not decided,
- Store privacy/deletion claims cannot be made honestly.
